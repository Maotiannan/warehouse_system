# 分段查询与结果恢复设计

## 1. 目标

为仓库查询桌面程序增加可持续维护的分段查询和本地恢复能力，同时修复唛头编辑保存、多个条件“或”筛选和查询区域空白问题。

本次功能不改变远程仓库接口，不新增后台服务，不修改已有远程数据。所有查询仍由桌面程序访问现有接口完成。

## 2. 已确认的业务规则

### 2.1 查询范围

- 不勾选长期查询：查询一个180天区间。
- 勾选“查一年”：查询最近360天，拆成2个连续且互不重叠的180天区间。
- 勾选“查两年”：查询最近720天，拆成4个连续且互不重叠的180天区间。
- 长期查询的终点固定为当天，不使用用户手工输入的结束日期。
- 日期区间含开始日和结束日。每段的开始日为该段结束日往前179天；下一段从上一段开始日前一天结束。
- 例如结束日为2026-07-20时，前两段是：
  - 2026-01-22至2026-07-20；
  - 2025-07-26至2026-01-21。
- “查一年”和“查两年”互斥，可同时都不选。
- 勾选长期查询时两个日期框显示总范围（360或720天）并禁用；取消后恢复可编辑并回到最新180天。
- 长期查询按“编号外层、时间段内层”顺序执行。多个编号和2/4个区间会分别请求，因此请求总数为“编号数 × 区间数”。
- 每个请求成功后立即追加结果并更新进度；失败后继续后续请求。
- 结果不去重，按请求完成顺序保留所有返回行。

### 2.2 恢复与持久化

- 只保存最后一次完整执行的查询快照。
- 完整查询返回0条也算成功，并覆盖旧快照。
- 任意分段失败时，本次结果可以继续在当前窗口显示，但不覆盖旧的完整快照。
- 快照包含所有返回字段、查询编号、状态、模式、实际分段、请求日志和响应日志。
- 重新打开程序时直接读取最后一次完整快照，不联网、不弹窗。
- 启动时按当前日期重新计算日期框：无模式为180天，查一年为360天，查两年为720天；保留长期查询勾选状态。
- 启动时清空高级筛选，显示快照中的全部原始结果。
- 启动时在结果上方显示一行紧凑提示，说明这是何时恢复的结果、实际查询范围和记录数。
- 新的完整查询成功后，用新快照原子替换旧快照；旧快照不能被半写入文件破坏。
- 快照损坏或版本未知时，记录诊断信息并以空结果启动，不弹窗阻塞使用。

### 2.3 唛头编辑

- 唛头仍然只允许编辑有稳定业务键的普通记录。
- 用户按回车、点击其他位置、切换行或关闭窗口时，当前编辑内容必须提交并保存。
- 不要求每次击键都写磁盘。
- 保存同时更新现有唛头缓存和最后一次查询快照中的对应记录，重启后显示修改后的值。

### 2.4 高级筛选

- 同一个筛选框内，英文逗号`,`和中文逗号`，`分隔多个条件，表示“或”。
- 不同筛选框之间表示“并且”。
- 唛头条件保持完全匹配；货主、进仓编号、作业号、货物名称、包装规格、车号、司机电话等条件保持包含匹配。
- 现有的空格、分号分隔兼容行为可以保留，但核心契约必须覆盖中英文逗号。
- 例如货主填`A,B`、车号填`沪A，沪B`，一行必须满足“货主为A或B”且“车号包含沪A或沪B”。

### 2.5 空白区域

查询进度和结果提示区域只在有内容时占用布局空间。空闲且没有提示时，外层容器同时隐藏并取消布局占位；查询遮罩仍覆盖表格显示进度。

## 3. 当前代码问题与边界

当前 `MainController` 直接串联输入、查询、过滤和展示；`WarehouseService` 每次查询会清空日志文件；`LocalEntryCacheService` 只保存单条记录和唛头，不保存查询结果快照。因此长期查询不能简单地重复调用现有方法，否则会丢失前段日志，也无法区分完整和部分成功。

本次只做针对性拆分，不重写远程接口解析、不改变现有列配置和导出格式。

## 4. 推荐架构

### 4.1 `QueryRangePlanner`

纯计算组件，输入结束日和查询模式，输出有序的`QuerySegment`列表。每个区间包含序号、总段数、开始日、结束日。它不访问 JavaFX、网络或文件，便于单元测试。

### 4.2 `QueryBatchService`

负责按编号和区间顺序调用一个可替换的单段查询客户端，并发布结构化进度事件。

建议接口边界：

```text
QueryBatchResult execute(QueryBatchRequest request, ProgressListener listener)
```

`QueryBatchResult`包含成功段、失败段、按完成顺序排列的结果、完整请求日志、完整响应日志和是否完整成功。网络客户端只负责一次“一个编号+一个180天区间”的请求；批次服务负责编排。

### 4.3 `QuerySnapshotStore`

负责最后一次完整快照的读取、写入和原子替换。文件放在现有用户目录：

```text
C:\Users\<当前用户>\.warehouse-query-system\query-snapshot.json
```

快照使用显式`schemaVersion`，保存查询元数据、分段清单、结果行和原始日志。写入流程为：写同目录临时文件、刷新并关闭、再用原子替换替换正式文件；失败时保留旧快照。

现有`entry-cache.json`继续负责唛头本地覆盖，避免把两种职责混在一个文件中。唛头保存成功后通过快照存储提供的“按稳定键更新当前快照”操作同步更新，不改变历史快照。

### 4.4 `AdvancedFilterMatcher`

把分词、字段匹配和AND/OR规则从控制器中抽出为纯组件。UI只提供字段文本，组件返回匹配结果和可测试的分词结果，确保所有筛选项遵循同一规则。

### 4.5 控制器职责

控制器只负责：读取控件值、切换查询模式、更新日期显示、订阅进度、追加结果、显示失败摘要、在完整成功后保存快照，以及在窗口关闭前提交活动中的唛头编辑。

## 5. 数据流

```text
控件输入
  -> QueryBatchRequest
  -> QueryRangePlanner 生成180天区间
  -> QueryBatchService 按编号/区间顺序执行
  -> 每段成功：UI追加结果、日志追加、进度更新
  -> 全部成功：QuerySnapshotStore 原子保存
  -> 有失败：UI保留成功结果并显示失败段，旧快照不变
```

启动流程：

```text
读取模式偏好 -> 根据今天重算日期框 -> 读取快照 -> 显示全部原始结果
             -> 清空高级筛选 -> 显示恢复提示
```

## 6. 错误处理

- 单段网络错误、解析错误或接口返回错误只标记该段失败，不中断后续编号和区间。
- 失败记录包含编号、段序号、开始日、结束日和可读错误信息。
- 所有段都失败时，当前窗口显示失败摘要；若存在旧完整快照，重启仍恢复旧快照。
- 所有段成功但返回0条时写入空结果快照。
- 程序关闭时若仍有查询任务，先停止提交新请求；已完成的部分不伪装成完整快照。
- 日志原文不截断，查询日志界面增加批次和分段标记，避免多次请求混在一起无法追查。

## 7. 自动化测试

不依赖远程服务、不依赖桌面窗口，优先通过可调用的服务接口测试：

1. `QueryRangePlanner`：验证180/360/720天、闰年、边界无重叠无遗漏。
2. `QueryBatchService`：使用假客户端验证编号/区间顺序、总请求数、逐段追加、部分失败继续、0条完整成功和日志聚合。
3. `QuerySnapshotStore`：验证原子写入、旧快照保护、完整快照读取、版本兼容和损坏文件降级。
4. `AdvancedFilterMatcher`：验证中英文逗号OR、字段之间AND、唛头完全匹配和其他字段包含匹配。
5. `MarkEditCoordinator`：验证回车、失焦、切换行和关闭前提交，并确认快照与唛头缓存同步。
6. Maven 编译、测试和打包作为发布门槛；所有测试使用临时目录和假网络客户端，不访问真实仓库接口。

## 8. 兼容与迁移

- 保留现有`entry-cache.json`格式和既有唛头数据。
- 新增快照文件不存在时按空结果启动，不影响首次运行。
- 保留现有`request_log.txt`和`response_log.txt`作为兼容输出，但由批次日志组件统一写入；启动时不再无条件删除上次完整日志。
- 不修改远程服务器数据，不改变已有照片、托信息、导出和列配置功能。

## 9. 完成标准

- 三种查询模式的日期显示、分段请求顺序和边界符合规则。
- 任意分段失败时后续请求继续，旧完整快照不被覆盖。
- 重启恢复结果无需联网且不弹窗；高级筛选为空；唛头修改可恢复。
- 查看日志能区分每个编号和每个时间段，并保留完整请求/响应内容。
- 高级筛选中英文逗号OR测试通过。
- 空闲状态不再出现无意义空白区域。
- 自动化测试、Maven构建、文档和Git状态全部通过检查。
